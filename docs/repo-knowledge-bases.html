Karpathy skills rationale

Repo Knowledge Bases for Coding Agents

Context, rationale, and differentiation for adding an OKF-compatible repo knowledge layer to the existing Karpathy audit and diff skills.

Generated: 2026-06-19 Audience: vibe coders and coding-agent tool builders Status: product rationale

Executive Summary

Google's Open Knowledge Format (OKF) already formalizes the lightweight "LLM wiki" pattern: a directory of Markdown files with YAML frontmatter, cross-links, optional index files, optional logs, and a permissive conformance model. We should not invent a competing file format.

The opportunity is to build a code-repo-specific discipline layer on top of OKF: producers that compile useful repo concepts, consumers that produce task briefs before coding, and guardrails that keep the wiki aligned with real code as diffs land.

Positioning: OKF is the storage and interchange format. A Karpathy repo knowledge-base skill is the coding-agent workflow that decides what to capture, when to consult it, how to verify it, and how to update it without drifting away from source code.

Background

The current repository contains three skills based on Andrej Karpathy's coding-agent principles:

  • /karpathy:audit reviews agent instruction files such as CLAUDE.md, AGENTS.md, and Cursor rules. It asks whether the instructions cover the four principles and whether the instruction file itself is lean, current, unambiguous, and verifiable.
  • /karpathy:diff reviews a code diff before commit. Its core test is whether every changed hunk traces to the task that was asked.

The audit and diff skills address two important failure points: bad standing instructions and bad generated diffs. The wiki MVP addresses the repeated orientation problem that happens before coding starts: every agent session re-reads the same files, rebuilds the same mental map, misses relationships across files, and spends tokens discovering context that should have accumulated.

Karpathy's LLM Knowledge Base pattern points at that missing layer. Raw material is stored as immutable source data, an LLM compiles it into a clean Markdown wiki, the wiki is maintained by agents rather than humans, and subsequent questions operate mostly over the compiled knowledge instead of the raw pile.

Problem for Vibe Coders

Vibe coders delegate a large share of implementation to an agent, but they still need the agent to understand the repo before it edits. The practical failure mode is not that the agent cannot read files. The failure mode is that it reads too many files shallowly, forgets relationships from prior sessions, and then makes plausible but wrong edits.

Cold Start Cost

Each new task burns tokens on repo discovery: file layout, naming conventions, common tests, APIs, data flow, and local invariants.

Context Fragmentation

Critical facts live in code, comments, tests, old PRs, docs, screenshots, and the developer's memory. Agents assemble that context inconsistently.

Untrusted Momentum

Once the agent starts editing from a weak mental model, the user has to detect scope creep, missed invariants, and silent assumptions after the damage is already in the diff.

The useful product for vibe coders is not a prettier generated wiki. It is a fast, local, inspectable orientation layer that answers: 

For this task:
- What files probably matter?
- What concepts and invariants should the agent read first?
- What source citations support those claims?
- What tests or commands prove success?
- What parts of the repo should the agent avoid touching?

Existing Landscape

Google Open Knowledge Format

OKF v0.1 is a draft open format for representing knowledge as Markdown files with YAML frontmatter. It requires a non-empty type field for each concept document, recommends fields such as title, description, resource, tags, and timestamp, supports standard Markdown links between concepts, and defines reserved index.md and log.md files for progressive disclosure and update history.

The spec explicitly avoids defining a fixed taxonomy, storage layer, serving model, SDK, or query infrastructure. That is its strength. It is a lingua franca, not a product workflow.

Google Knowledge Catalog Reference Work

Google's announcement and repository ship reference implementations: a BigQuery-oriented enrichment agent, sample bundles, and a static HTML visualizer. These make OKF concrete, but they are proofs of concept and are weighted toward data catalog use cases.

Graphify-Style Knowledge Graph Tools

Graphify-style tools point at a folder and produce an interactive graph, a report, and a queryable graph artifact across code, docs, PDFs, images, and other materials. This is close to the codebase orientation problem and may overlap with repo map generation.

The important distinction is that a graph generator is usually optimized for extraction and query. The Karpathy repo knowledge layer is optimized for disciplined coding-agent behavior before, during, and after edits.

ByteRover-Style Agent Memory

ByteRover-style tools focus on portable, inspectable memory for coding agents. Their framing is strong: file-based memory can be diffed, branched, merged, inspected, and shared without a hidden vector database.

This layer should learn from that design, but stay narrower: repo-local, OKF-compatible, task-brief-first, and integrated with Karpathy audit and diff review.

Traditional RAG and Vector Search

RAG systems can be useful at larger scale, but they often introduce infrastructure before the repo needs it. Karpathy's observation is that at small and medium scale, an agent can do surprisingly well with maintained indexes, summaries, and links. That is especially true for repos where the knowledge can live beside the code in version control.

What We Are Building

The third Karpathy skill is /karpathy:wiki. The skill maintains an OKF-compatible repo knowledge bundle and uses it to produce implementation briefs before coding. It should make agents more deliberate, more surgical, and easier for vibe coders to supervise.

Implementation status: v1.2.0 ships the first MVP: wiki scaffold, manifest generation, search, doctor checks, and advisory stale reminders. Semantic concept writing, task briefs, Q&A, and surgical wiki updates are LLM-led workflows guided by the skill; the helper script intentionally does not try to understand arbitrary repos by itself.

Source code should not be copied into raw/. The repository is already the source. Wiki concepts should cite and link back to code, tests, docs, and external material.

Repo Concept Types

OKF does not prescribe concept types, so the skill should define a small repo-specific vocabulary that agents can use consistently:

Type Purpose Typical Resource
Component A coherent implementation area with ownership, dependencies, and public surface. Source directory or primary module.
Workflow A user or system flow that crosses files and services. Routes, handlers, tests, or product docs.
Invariant A rule that must remain true across changes. Tests, comments, validation code, schema constraints.
Decision A durable architectural or product decision, with context and tradeoffs. ADR, PR, issue, commit, or cited code.
Test Surface The commands and files that verify a behavior area. Test files, package scripts, CI config.
Task Recipe A reusable implementation path for common tasks. Prior diffs, docs, and tests.
Failure Mode A known bug pattern, sharp edge, or operational risk. Issues, test regressions, logs, runbooks.

Command Surface

The public command surface should stay small. Users should not need to learn the wiki lifecycle. The skill should infer setup, refresh, briefing, answering, updating, and health checks from repo state and from the user's input.

Command When a User Enters It What the Skill Does
/karpathy:wiki
/karpathy wiki		 No arguments, or the user wants the wiki to decide what is needed. Context-aware entrypoint: setup if no wiki exists, update if code changed, status if clean.
/karpathy:wiki how does auth work?
/karpathy wiki how does auth work?		 The input is question-shaped. Answer from the wiki first, then read cited source files if needed. Return citations and stale warnings.
/karpathy:wiki add trial expiration handling
/karpathy wiki add trial expiration handling		 The input is task-shaped. Produce a task brief: likely files, relevant concepts, invariants, risks, and verification commands.
/karpathy:wiki doctor
/karpathy wiki doctor		 The user explicitly wants maintenance. Run a health check for broken links, stale source references, uncited claims, and OKF conformance issues.

End-to-End User Journeys

The guiding UX rule is that the user remembers one command and the skill handles mode selection. The command accepts the colon form and documents the human-friendly space form: /karpathy:wiki and /karpathy wiki.

First Run

User Enters Skill Returns Behind the Scenes
/karpathy wiki "No repo wiki found. I can create one at knowledge/wiki/ using OKF-compatible Markdown. I will start with repo overview, component map, test surfaces, known commands, index.md, and log.md. Proceed?" Detect missing bundle, explain the target files, and ask before writing.

Setup Automation

User Enters Skill Returns Behind the Scenes
yes to setup "Automation level: 1. Manual only. 2. Remind me when the wiki is stale. 3. Strict stale reminder that blocks commits until the wiki is reviewed." Offer automation as a setup choice while keeping semantic wiki edits in the normal agent approval flow.

Coding Task

User Enters Skill Returns Behind the Scenes
/karpathy wiki add trial expiration handling A task brief with likely files, relevant concepts, invariants, risks, and verification commands. It ends with: "Before coding, I will read the cited source files. Proceed?" Classify the input as a task, read indexes and relevant concepts, then require source verification before edits.

Repo Question

User Enters Skill Returns Behind the Scenes
/karpathy wiki how does auth work? A cited answer grouped by relevant concepts and source files, plus any stale claims that should be refreshed. Classify the input as a question, answer from wiki first, then read cited source where needed.

Changed Code

User Enters Skill Returns Behind the Scenes
/karpathy wiki "I found code changes since the wiki was last updated. Affected concepts: Billing Lifecycle and Billing Test Surface. Apply proposed wiki updates?" Inspect git diff, map changed files to concepts by citations and resources, and propose only affected wiki edits.

Explicit Health Check

User Enters Skill Returns Behind the Scenes
/karpathy wiki doctor A wiki health report: critical issues, warnings, clean checks, and proposed fixes. The skill asks before applying changes. Run OKF conformance checks, link checks, citation checks, stale source checks, and consistency checks.

Automation Model

The skill should make the useful path automatic without hiding source-of-truth changes from the user. Setup should offer automation levels and explain the tradeoff plainly.

Automation Levels

Level Behavior Best For
Manual only No hooks. The user or agent runs /karpathy wiki. Repos that want zero Git hook behavior.
Remind me when stale Install non-mutating checks that print an actionable reminder when changed files affect wiki concepts. Most users who want automation without files changing behind their back.
Strict stale reminder Block commits when changed files affect wiki concepts, but still do not edit files automatically. Repos that explicitly prefer enforcement over advisory reminders.
Agent-initiated update /karpathy:wiki detects stale concepts, proposes edits, asks approval, and updates only affected pages. Most teams that want useful automation without hiding semantic edits in Git hooks.

How Stale Reminders Work

"Remind me when stale" should be a non-mutating safety net. It should never edit files, stage files, or block commits by default. It only detects that the wiki is probably stale and tells the user what to run.

  1. Maintain a lightweight manifest such as knowledge/wiki/.karpathy-wiki.json mapping source paths to concept files, based on resource fields and citations.
  2. Install a Git hook, preferably pre-commit, that computes the staged source paths with git diff --cached --name-only.
  3. Compare staged paths to the manifest. If no changed source file is referenced by the wiki, print nothing and allow the commit.
  4. If changed files map to wiki concepts and those concepts are not staged, print a concise reminder naming the affected concepts and the command to run.
  5. Exit successfully by default so the hook is advisory. A repo can opt into strict mode if it wants stale wiki checks to block commits, but the hook still must not edit files.

- Billing Lifecycle: knowledge/wiki/workflows/billing-lifecycle.md
- Billing Test Surface: knowledge/wiki/tests/billing.md

Agent Instruction Setup

During setup, the skill should detect AGENTS.md, CLAUDE.md, and tool-specific rule files, then offer to add a compact block so future agents use the wiki without the human remembering to ask. It should never stage or commit those instruction files automatically; the human decides whether they belong in version control. 

## Repo Wiki

Use /karpathy:wiki before non-trivial code changes to orient on relevant components,
invariants, tests, and risks. You can also type "karpathy wiki" as plain text.
If code changes affect documented concepts, update the affected wiki pages before
finishing. Treat the wiki as advisory: read cited source files before editing,
and keep changes surgical.

Differences From Existing Tools

System What It Provides Gap for This Use Case What We Add
OKF Format Markdown plus YAML frontmatter, links, indexes, logs, and conformance rules. Does not define a code-repo taxonomy, coding workflow, or update discipline. Use OKF as the substrate and define repo-specific producers, consumers, concept types, and maintenance rules.
Google Knowledge Catalog samples Reference implementation BigQuery enrichment agent, sample bundles, and visualizer. Weighted toward data catalog enrichment, not source-code editing loops. Apply the pattern to code repos, tests, invariants, task recipes, and diff-aware maintenance.
Graphify-style tools Extraction and graph Folder-to-graph conversion, graph report, queryable artifact. May optimize for mapping and Q&A more than disciplined implementation behavior. Task briefs, source verification, Karpathy principles, and integration with audit/diff review.
ByteRover-style memory Agent memory Inspectable local memory for coding agents. Broader memory layer, not necessarily an OKF repo wiki tied to a specific diff loop. Repo-local OKF bundle with task-first retrieval and surgical updates driven by changed files.
Existing /karpathy:audit Instruction hygiene Reviews standing agent instructions. Does not compile or query repo knowledge. Wiki rules can be audited; audit can ensure the agent is told how to use the wiki.
Existing /karpathy:diff Change hygiene Reviews whether a diff traces to the task. Works after edits; does not reduce bad orientation before edits. Wiki briefs improve pre-edit orientation; diff review can require affected wiki concepts to be updated.
Human-written docs Manual context Useful when current, concise, and trusted. Humans abandon documentation upkeep because it is repetitive bookkeeping. Agents maintain links, indexes, summaries, and logs as part of the coding workflow.
Traditional RAG Retrieval infrastructure Scales search and retrieval across large corpora. Can add hidden infrastructure and opaque ranking before a repo needs it. Start with inspectable files and indexes; add search only when the bundle outgrows simple traversal.

Design Principles

The repo knowledge-base skill should apply the same four Karpathy principles to knowledge management.

Karpathy Principle Wiki Interpretation Practical Rule
Think Before Coding Orient before editing. For non-trivial tasks, produce a task brief from the wiki before implementation.
Simplicity First Use simple files before infrastructure. Markdown, YAML, links, indexes, logs. No vector database or graph database in the default path.
Surgical Changes Update only affected knowledge. After a diff, identify changed concepts and touch only those wiki files.
Goal-Driven Execution Knowledge must support verification. Concepts should cite source files and name the commands that verify the behavior area.

Core Behavioral Contract

  1. The wiki is advisory. Source code, tests, and explicit user instructions are authoritative.
  2. A concept claim that affects implementation must have a citation to source material.
  3. The agent may use the wiki to decide where to look, but it must read cited source before editing.
  4. The agent should not copy large source files into the wiki. It should summarize and link.
  5. Staleness is allowed only when visible: concepts should carry timestamps, source commit hints, confidence, and verification commands where practical.

MVP Scope

The first useful version should be small. It should not try to become a universal repo graph engine on day one.

Must Have

  • Create knowledge/wiki/ as an OKF-compatible bundle.
  • Generate a root index.md and log.md.
  • Compile concepts from existing docs, tests, package scripts, and obvious source entry points.
  • Use a fixed initial concept vocabulary: Component, Workflow, Invariant, Decision, Test Surface, Task Recipe, and Failure Mode.
  • Produce a task brief that lists likely files, related concepts, risks, and verification commands.
  • After changes, update only concepts tied to changed files.
  • Run a lint pass for broken links, missing type, missing citations for strong claims, and stale source references.

Should Not Have Yet

  • No hidden hosted service.
  • No mandatory embeddings.
  • No custom graph database.
  • No promise that the wiki replaces source-code reading.
  • No massive generated encyclopedia that the agent loads wholesale every turn.

- src/api/webhooks/stripe.ts

- [Billing Lifecycle](../wiki/workflows/billing-lifecycle.md)

- Webhook handlers must be idempotent.
- Entitlements are derived from subscription state.
- Raw Stripe request body must be preserved for signature verification.

## Verification
- npm test -- billing
- npm run typecheck

## Notes
- Read the cited source files before editing. The wiki narrows the search; it is not the authority.

Risks and Guardrails

Risk Why It Matters Guardrail
Wiki drift The agent trusts stale summaries and edits against old assumptions. Store timestamps and source commit hints; lint changed files against affected concepts; require source reads before edits.
Doc bloat The wiki becomes another token sink and obscures useful facts. Prefer concise concepts, indexes, and task briefs. Do not duplicate full source files.
False authority Generated prose can sound more certain than the code supports. Require citations for implementation-relevant claims and include confidence fields for inferred knowledge.
Scope creep The skill becomes a generic knowledge platform instead of a coding aid. Keep MVP focused on repo orientation, task briefs, diff-aware updates, and linting.
Competing with OKF A custom format would reduce portability and duplicate the emerging standard. Emit conformant OKF and add only producer-defined fields allowed by the spec.

Product Bet

The core bet is that vibe coders do not need another documentation site. They need a local, inspectable repo brain that turns accumulated agent work into better future agent work.

The first-order benefit is reduced token waste and faster orientation. The deeper benefit is behavioral: agents start from a task brief grounded in repo-specific invariants, then end by updating the knowledge layer and passing a diff review. That creates a loop where exploration and implementation compound instead of evaporating between sessions.

Thesis: The valuable product is not "generate Markdown from a repo." The valuable product is "make every future coding-agent session start with the repo knowledge learned by the previous sessions, while keeping that knowledge cited, local, and diffable."
